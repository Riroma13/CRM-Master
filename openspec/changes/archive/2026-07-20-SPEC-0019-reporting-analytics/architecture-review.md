# Architecture Review — SPEC-0019: Reporting & Analytics

**Verdict: REJECTED**

## Findings

| # | Severity | Topic | Finding | Effort | Recommendation |
|---|----------|-------|---------|--------|----------------|
| 1 | 🔴 Blocking | KPI formula injection | `mathjs.evaluate(formula)` on user-provided string is RCE. No sandbox, no allowlist, no AST validation. | High | Replace `mathjs.evaluate` with a safe expression parser (e.g., `expr-eval` with restricted scope, or a custom DSL that only allows `+ - * / ( )` and known metric names). |
| 2 | 🔴 Blocking | No read-only enforcement | Design says "never queries operational tables" but has zero enforcement — no Prisma middleware, no separate DB role, no trigger, no middleware intercepting cross-model JOINs. SPEC-0018 got 3 layers; SPEC-0019 gets 0. | High | Add Prisma middleware that blocks any query model outside `AnalyticsDataset`, `AnalyticsSnapshot`, `KPI`, `Dashboard`, `DashboardWidget`, `ReportDefinition`, `ReportExecution`, `ExportJob`. Document the explicit allowlist. |
| 3 | 🔴 Blocking | No DatasetEvent producer exists | `DatasetEvent` is defined as a shared contract but no module in the codebase implements `DatasetPublisher` or emits `DatasetEvent`s. The ingestion pipeline has no source. Without this, the entire platform is dead on arrival — no data ever reaches AnalyticsDataset. | High | Before Apply, produce a module-by-module inventory: (a) which modules will emit DatasetEvents, (b) what metrics each produces, (c) implementation order. Wire the first producer (e.g., Workflow Engine → workflow.created count) before the ingestion pipeline goes live. |
| 4 | 🔴 Blocking | Prisma does not support PARTITION BY | Design describes partitioning by tenant + month across granularities. Prisma `schema.prisma` does not support `PARTITION BY` — the model `@@map("analytics_datasets")` creates a flat table. The partitioning strategy is aspirational SQL that cannot be expressed in the schema. | Medium | Move partitioning to raw SQL migration scripts (same pattern as SPEC-0018). The `schema.prisma` model is the logical definition; the actual table is created via `CREATE TABLE analytics_datasets (...) PARTITION BY RANGE (window_start)`. Document that partition management is done outside Prisma. |
| 5 | 🔴 Blocking | Export files have no access control | `ExportJob.filePath` is a string with no storage backend specification, no path scheme, and no tenant verification at download time. Any user who discovers or guesses another tenant's export file path can download it. No tenant-scoped download endpoint exists. | High | (a) Store export files in tenant-scoped paths: `exports/{tenantId}/{jobId}.pdf`. (b) Serve via a controlled endpoint: `GET /api/v1/reporting/exports/{jobId}/download` that validates tenantId from auth context. (c) Never expose raw file paths. |
| 6 | 🔴 Blocking | Lost events are undetectable | Dataset ingestion uses UPSERT by unique key — if an event is lost (BullMQ failure, network issue, worker crash), the dataset silently misses the update. No checksum, no expected-event-count, no periodic reconciliation. A lost event is invisible forever. | High | (a) Add a `DatasetIngestionLog`: append-only log of every event received with status. (b) Add periodic reconciliation: expected vs actual counts per dataset per window. (c) Add a replay endpoint to re-process a time window. |
| 7 | 🟡 High | Granularity unmanageable | Triple storage per metric (hourly 7d, daily 12mo, monthly 3yr) means every metric event is written 3× to different partitions. A metric with 10K events/day generates 30K writes. Storage amplification without query-time dedup. | Medium | Store only daily granularity by default. Compute hourly from daily (window functions) and monthly from daily (aggregation). Only pre-materialize other granularities when a specific report requires it. |
| 8 | 🟡 High | Formula sandbox escape | Even with `expr-eval`, an expression like `workflows_created / 0` causes an unhandled error that crashes the KPI computation. No error isolation per KPI. One bad formula breaks all KPIs. | Medium | Each KPI evaluation must be isolated (try/catch per KPI). A failing KPI returns `status: 'error'` without affecting other KPIs. Log the error for the tenant admin. |
| 9 | 🟡 High | Dashboard N+1 widget queries | A dashboard with 20 widgets triggers 20 separate cache lookups and 20 potential MISS→COMPUTE cycles. No batch loading. No hydrator that resolves all widget data in a single plan. | Medium | Add a `DashboardHydrator`: given a dashboard layout, compute all widget data in parallel (Promise.all) with a single cache multi-get. If all MISS, batch the dataset queries. |
| 10 | 🟡 High | Retention conflicts with historical analysis | Hourly data kept 7 days, daily 12 months. A report comparing "this year vs last year" for hourly trends would fail silently — last year's hourly data is gone. No strategy for long-term trend analysis across granularity boundaries. | Medium | Materialize monthly aggregates that survive the full 3-year retention. Document that sub-monthly granularity is only available for the last 12 months. Provide an API that automatically degrades granularity based on available data. |
| 11 | 🟡 High | Refreshing snapshots invalidates all widgets | Design says "Mark related snapshots as stale" on any dataset update. If a frequently-updated metric triggers snapshot invalidation 100×/day, all dashboard widgets for that dataset recompute simultaneously. Cache stampede. | Medium | Use TTL-based expiration, not event-driven invalidation. Snapshots expire after TTL regardless of updates. Add a "stale-while-revalidate" pattern: serve stale data + trigger async refresh. |
| 12 | 🟡 High | Missing tenantId in ReportExecution and ExportJob schemas | `ReportExecution` (Prisma model) does not include a `tenantId` field. Future cross-tenant queries or support/admin access cannot scope execution records. Same gap in `DashboardWidget` and `AnalyticsSnapshot`. | Low | Add `tenantId` to all models that lack it. Every model in the reporting bounded context should have tenantId for consistency. |
| 13 | 🟡 High | No `@@index([tenantId])` on some models | `Dashboard`, `ReportDefinition`, `DashboardWidget`, `ReportExecution` do not have explicit `@@index([tenantId])`. Querying all dashboards for a tenant requires a full scan. | Low | Add `@@index([tenantId])` on all models that have `tenantId` but no composite index starting with it. |
| 14 | 🟡 High | Exporter interface lacks context | `Exporter.export(data, options)` receives raw data but has no tenant context, user context, or request context. Error reporting, audit logging, and format-specific decisions cannot access tenant config. | Low | Add `ExportContext` with `tenantId`, `userId`, `format`, `options`, and `correlationId`. Pass through the entire export pipeline. |
| 15 | 🟡 High | No `DatasetPublisher` interface in shared contracts | Design defines `DatasetEvent` as a type but doesn't include a `DatasetPublisher` interface that modules would implement. Without it, each module invents its own publishing pattern. | Medium | Add `DatasetPublisher.publish(event: DatasetEvent): Promise<void>` to the shared contracts. Same pattern as `AuditPublisher` (SPEC-0018) and `ActivityPublisher` (SPEC-0017). |
| 16 | 🟢 Nice | PDF export engine unspecified | Open Question #4: `jsPDF` vs `Puppeteer`. jsPDF is not HTML-to-PDF — it's a programmatic canvas. Cannot render existing dashboard HTML. Puppeteer requires Chrome binary on the server. | Low | Recommend Puppeteer for dashboard PDF (headless Chrome renders the actual UI). Use jsPDF for data-only tabular reports. |
| 17 | 🟢 Nice | TTL granularity mismatch | KPI cache TTL is per-KPI in seconds. Dataset snapshot TTL is per-snapshot in seconds. A dashboard with both KPI widgets and chart widgets will have different staleness for different widgets on the same screen. | Low | Recommend a dashboard-level `refreshInterval` that overrides individual widget TTLs. UI can then use a single refresh timer. |
| 18 | 🟢 Nice | No backfill plan | Event-driven ingestion only captures events from the moment the pipeline starts. Historical data (before SPEC-0019 deployment) never enters AnalyticsDataset. Reports showing "all time" will be empty for past periods. | Low | Add a one-time backfill task: a script that queries operational tables ONCE (with EXPLICIT approval and monitoring) to seed the datasets, then the event pipeline takes over. |

## Risks

| Risk | Unmitigated? | Reality |
|------|-------------|---------|
| KPI formula RCE | **YES — BLOCKING** | `mathjs.evaluate` with tenant-controlled input is a remote code execution vector. |
| No data source | **YES — BLOCKING** | DatasetEvent contract exists but zero modules produce it. The pipeline has no data to ingest. |
| Export file access | **YES — BLOCKING** | No tenant scoping, no controlled download endpoint. Path traversal or guess attack. |
| Silent data loss | **YES — BLOCKING** | UPSERT silently accepts any value. Lost events are invisible forever. |
| Read-only is aspirational | **YES — BLOCKING** | No enforcement mechanism. A future developer can add an operational table JOIN without any guard stopping them. |

## Conditions for re-submission

1. **Replace `mathjs.evaluate` with safe expression parser** (e.g., `expr-eval` with restricted scope). Add per-KPI error isolation.
2. **Add read-only enforcement**: Prisma middleware allowlisting only reporting models. Separate DB role.
3. **Produce a DatasetEvent producer inventory** before Apply: which modules, what metrics, implementation order.
4. **Move partitioning to raw SQL migrations** — document that Prisma schema is the logical definition only.
5. **Add tenant-scoped export storage + controlled download endpoint** with auth verification.
6. **Add DatasetIngestionLog + periodic reconciliation + replay endpoint** for lost event detection.
7. **Add `DatasetPublisher` interface** to shared contracts.
8. **Fix missing `tenantId` and `@@index([tenantId])`** on all reporting models.
9. **Add DashboardHydrator** for batch widget data resolution.
10. **Fix granularity storage amplification** — default to daily, compute others via window functions.
